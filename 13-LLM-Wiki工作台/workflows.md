# LLM-Wiki工作流

## 输入分流工作流

当用户提供零散想法时，先判断以下字段：

| 字段 | 示例 |
|---|---|
| 主题 | 广告、标题、五点、定价、库存、竞品、售后、合规 |
| 状态 | 已验证、待验证、个人判断、原始资料、待确认事实 |
| 适用范围 | 全店、某尺寸、某 ASIN、某颜色、某广告活动 |
| 证据 | 报表、后台截图、订单表现、竞品观察、客服反馈、经验判断 |
| 风险 | 合规、误导、库存、利润、广告浪费、售后预期 |

## 写入位置规则

| 输入类型 | 主写入位置 | 同步更新 |
|---|---|---|
| 已确认产品事实 | `02-产品事实库` | `13-LLM-Wiki工作台/index.md` |
| 待确认产品事实 | `12-收集箱与附件/03-待确认产品信息.md` | 相关模块增加待确认提醒 |
| 已验证运营结论 | `09-规则案例与实验/06-已验证结论` | 对应业务模块总览、`index.md`、`log.md` |
| 待验证运营假设 | `09-规则案例与实验/01-待验证假设` | `13-LLM-Wiki工作台/wiki/问题队列/待补证据清单.md` |
| 原始资料 | `13-LLM-Wiki工作台/raw/对应主题` | 需要时编译到业务模块 |
| 跨模块策略 | `13-LLM-Wiki工作台/wiki/策略地图` 或 `wiki/跨模块结论` | 相关业务模块入口 |
| 售后话术 | `07-售后客服与合规` | 合规规则、待确认信息 |

## 已验证结论编译步骤

1. 保留用户原始结论。
2. 写明适用条件。
3. 写明操作规则。
4. 写明复盘指标。
5. 写明风险边界。
6. 写明待补数据。
7. 在业务模块总览或记录表中建立入口。
8. 更新 `13-LLM-Wiki工作台/index.md` 和 `log.md`。

## 待验证假设编译步骤

1. 不写成规则。
2. 写成可测试假设。
3. 标注验证指标和观察周期。
4. 标注最小实验动作。
5. 标注停止条件。
6. 放入待验证目录或待验证观点。

## 查询回答工作流

回答用户运营问题时：

1. 先查 `13-LLM-Wiki工作台/index.md`。
2. 再查对应业务模块总则。
3. 如果涉及产品事实，必须查 `02-产品事实库/00-产品矩阵总表.md` 和禁区规则。
4. 如果涉及售后或评价，必须查 `07-售后客服与合规/00-售后总原则.md`。
5. 如果信息不足，明确标注 `待确认` 或 `待验证`。

## 原始资料命名建议

| 类型 | 文件名 |
|---|---|
| 用户想法 | `YYYY-MM-DD-主题-原始想法.md` |
| 广告报表摘录 | `YYYY-MM-DD-广告-报表摘录-对象.md` |
| 定价观察 | `YYYY-MM-DD-定价-观察-对象.md` |
| 竞品观察 | `YYYY-MM-DD-竞品-观察-ASIN或品牌.md` |
| 售后案例 | `YYYY-MM-DD-售后-问题类型-ASIN.md` |
