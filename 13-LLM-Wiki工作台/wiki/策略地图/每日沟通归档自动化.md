## 去重规则

归档前必须先检查：

- `13-LLM-Wiki工作台/log.md`
- 相关业务模块
- 已验证结论
- 待验证假设
- 待补证据清单
- 更新记录

如果同一主题、同一结论或同一原始资料已经入库，只在汇报中说明已入库并跳过，不重复新增页面、不重复追加日志、不重复改写索引。

## 写入规则

仅对尚未入库的新内容进行整理，并按以下类型写入：

| 类型 | 写入位置 |
|---|---|
| 已确认事实 | `02-产品事实库` |
| 已验证结论 | `09-规则案例与实验/06-已验证结论` |
| 待验证假设 | `09-规则案例与实验/01-待验证假设` 或 `12-收集箱与附件/04-待验证观点.md` |

| 待确认产品信息 | `12-收集箱与附件/03-待确认产品信息.md` |
| 待补证据 | `13-LLM-Wiki工作台/wiki/问题队列/待补证据清单.md` |
| 决策日志 | `10-决策日志与复盘` |

## 输出要求

每次归档后在当前线程简短汇报：

- 新增整理了哪些主题。
- 跳过了哪些已入库主题。
- 更新了哪些文件。
- 还需要用户补充哪些证据。

如果没有亚马逊相关新内容，应说明未发现需归档的新内容。

## Git 更新、提交与推送规则

归档开始前自动化应先更新 Git 仓库：

- 进入 `/Users/kedaya007/Documents/Obsidian/亚马逊罗马伞`。
- 先执行 `git status --short`。
- 如果存在未提交改动，暂停本次自动归档，汇报本地变更文件，不强制拉取、不覆盖本地内容。
- 如果工作区干净，执行 `git pull --ff-only origin main`。
- 如果拉取失败、出现冲突或远端不是 fast-forward，停止后续归档并汇报失败原因，不执行会覆盖本地内容的命令。

归档完成后自动化应再次检查 `git status --short`：

- 如果没有新内容或没有文件变更，不创建空提交、不拉取后提交、不推送。
- 如果本次归档产生了知识库文件变更，执行 `git add -A`。
- Commit message 使用 `daily archive: YYYY-MM-DD`，日期使用当天中国日期。
- 提交成功后执行 `git push origin main`。
- 如果提交或推送因为认证、网络、远程领先或冲突失败，不强推、不重置，保留本地变更并在当前线程汇报失败原因和需要用户处理的动作。

## 风险边界

- 不得编造 ASIN、SKU、尺寸、认证、防风等级、底座重量、质保承诺、Power Bank 等未确认信息。
- 不确定内容必须标记为待确认或待验证。
- 如果用户已经要求即时入库，自动化不得二次归纳同一内容。
