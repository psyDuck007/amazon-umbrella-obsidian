# LLM-Wiki工作台

## 定位

本目录用于把零散输入、已验证经验、待验证假设和跨模块复盘，按照 Karpathy 的 LLM Wiki 方法沉淀成可长期维护的运营知识库。

它不是替代现有业务目录，而是知识库的工作台：

- `raw/`：原始输入层，保留用户想法、报表摘录、竞品观察、客服案例等未经整理资料。
- `wiki/`：编译知识层，沉淀跨模块概念、策略地图和复用结论。
- `index.md`：全库索引，帮助 AI 和人工快速找到相关页面。
- `log.md`：编译日志，记录每次新增、整理、合并和废弃的知识动作。
- `lint.md`：健康检查清单，用于定期发现矛盾、重复、孤立页面和待补数据。

## 核心原则

1. 原始资料不直接覆盖已确认事实。
2. 已确认产品事实仍以 `02-产品事实库` 为准。
3. 已验证运营经验进入 `09-规则案例与实验/06-已验证结论`，并在相关业务模块建立入口。
4. 待验证想法进入 `09-规则案例与实验/01-待验证假设` 或 `12-收集箱与附件/04-待验证观点`。
5. 涉及 ASIN、SKU、尺寸、底座、认证、防风、质保、配件、Power Bank 等事实，未确认前不得写成事实。

## 推荐工作流

### 1. 收集

用户可以用自然语言输入：

```text
广告优化 已验证 xxx
定价策略 待验证 xxx
标题优化 个人判断 xxx
竞品观察 原始资料 xxx
```

AI 先判断主题、状态、适用范围、证据和风险，再选择写入位置。

### 2. 编译

AI 将输入拆成：

- 已确认事实
- 已验证结论
- 待验证假设
- 操作 SOP
- 案例复盘
- 禁止事项
- 待补数据

然后更新对应业务模块、跨模块 wiki 页面、索引和日志。

### 3. 查询

回答运营问题时，AI 应优先读取：

1. `13-LLM-Wiki工作台/index.md`
2. `00-知识库入口/00-知识库导航.md`
3. 相关业务模块总则
4. 具体规则页或案例页

### 4. 维护

定期执行 `lint.md` 检查：

- 是否存在互相矛盾的结论。
- 是否有已验证结论没有入口。
- 是否有待确认事实被误写成 Listing 或客服话术。
- 是否有孤立页面没有被索引。
- 是否有实验缺少复盘指标。

## AI接入使用

知识库可以作为 AI 运营助手的知识源，用于问答、报表分析、SOP 执行、自动整理、决策辅助和周/月复盘。详细规则见 [[13-LLM-Wiki工作台/wiki/策略地图/AI接入与运营助手使用方式]]。

当前线程已配置每日沟通归档自动化，用于把尚未入库的亚马逊相关沟通内容整理进知识库。详细规则见 [[13-LLM-Wiki工作台/wiki/策略地图/每日沟通归档自动化]]。
